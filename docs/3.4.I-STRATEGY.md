# 3.4.I — Hybrid-Search Strategie

**Stand:** 13. Mai 2026, Nachmittag (Tag 14). Nach 3.4.A–H abgeschlossen,
nach End-to-End-Smoke mit substantiellem Halluzinations-Befund.

Auslöser: Phase-3.4-Smoke-Test (siehe `docs/3.4-SMOKE.md`) hat gezeigt,
dass Pure Vector-Search mit Threshold 0.7 bei deutschen Eigennamen-Queries
Token-Overlap-Hits produziert, aus denen der LLM detaillierte
Memory-Halluzinationen rekonstruiert. Konkretes Beispiel: Query
"Bayreuth-Karten und Marc" liefert 3 Hits mit Top-Sim 0.704, aber keiner
ist thematisch relevant. Twin halluzinierte daraufhin eine detaillierte
fiktive Diskussion mit vier Punkten.

3.4.I adressiert das via Hybrid-Search (Vector + FTS5) plus Prompt-
Wording-Update. 3.4.J (geplant) ergänzt LLM-Re-Rank + Anti-Halluzinations-
Pattern.

## Architektur-Entscheidungen

### 1. Ranking-Strategie: Reciprocal Rank Fusion (RRF)

**Wahl:** RRF mit `k=60` als Standard-Konstante.

```
final_score(item) = 1/(k + vector_rank) + 1/(k + bm25_rank)
```

Items, die nur in einer Source auftauchen, kriegen für die fehlende Liste
Rank = unendlich (Score-Beitrag = 0). Implizit bevorzugt RRF damit Items,
die in beiden Sources ranken — was natürlich gegen Halluzinations-Risiko
arbeitet (Token-Overlap-Hits ohne semantische Match landen niedriger).

**Verworfen:**

- **Linear Combination (α × vector + (1-α) × bm25):** Hyperparameter-
  Tuning, BM25-Normalisierungs-Probleme, magic-α.
- **LLM-Re-Rank:** zusätzlicher LLM-Call pro Send, +1-3s Latenz, eigenes
  Halluzinations-Risiko. Kommt als 3.4.J nach Phase-5-Validierung.

**Begründung:** RRF ist Industrie-Standard (Elastic, Vespa, Pinecone),
rang-basiert daher score-skalen-unabhängig, single-Sub-Schritt-machbar.
Bei Phase-5-Validierung wird gemessen, ob RRF allein reicht oder ob
LLM-Re-Rank zusätzlich nötig wird.

### 2. Query-Vorverarbeitung: Aggressive Sanitization

**Wahl:** vor FTS5 alle Nicht-Buchstaben/Nicht-Zahlen → Space, dann
Whitespace-Normalisierung.

```typescript
function sanitizeForFts5(query: string): string {
  return query
    .replace(/[^\p{L}\p{N}\s]/gu, ' ')  // Operatoren raus
    .replace(/\s+/g, ' ')
    .trim();
}
```

Beispiele:
- `"Markus' Bayreuth-Reise?"` → `Markus Bayreuth Reise`
- `"@florian, was meinst du?"` → `florian was meinst du`
- `"Workshop-Vertrag"` → `Workshop Vertrag`

**Min-Token-Filter:** wenn nach Sanitization weniger als 2 Tokens übrig,
FTS5-Search überspringen, nur Vector-Pfad nutzen. Verhindert FTS5-Aufruf
mit leerer/einzelwortiger Query.

**Vector-Pfad bleibt unverändert** — User-Message direkt embedden mit
E5-Prefix `query:` wie in 3.4.E.

**Verworfen:**

- **FTS5-Phrase-Query-Konstruktion (`"foo" OR "bar"`):** zu komplex,
  Stopword-Probleme, bei kurzen Queries absurd.
- **Try-Catch-Fallback:** Code-Smell, Bindestriche brechen *immer* —
  besser explizit handhaben.

**Begründung:** Robustheit > Eleganz. Deutsche Bindestrich-Komposita
sind häufig in Markus-Konversationen (`September-Workshop`,
`Discovery-Call`, `AI-Literacy`). FTS5-Tokenizer `unicode61
remove_diacritics 2` macht Lowercase intern.

### 3. Pool, Merge, Threshold

**Pool-Größe pro Source:** Top-10. Bei aktueller Datenlage (24 Conv)
ein Drittel, bei Production-Scale (~Hundert+) handhabbar. Sweet-Spot
zwischen RRF-Robustheit und Performance.

**Missing-Rank-Penalty (Pattern 1):** Items in nur einer Source kriegen
für die fehlende Liste Rank = ∞ → Score-Beitrag = 0. Effekt: Items in
beiden Sources werden implicit geboostet. Pattern verworfen wurde
"strict mode" (nur Items in beiden Sources behalten), weil das legitime
Fall-A-Hits killt (semantische Hits ohne Token-Match — z.B. "Wer ist
meine Frau?" findet "Anna" via Semantik, aber FTS5 findet's nicht weil
"Frau" ≠ "Anna" als Token).

**Threshold-Sicherung — zweistufig:**

1. **Pre-RRF Min-Vector-Sim:** Vector-Hits mit Sim < 0.5 kommen gar nicht
   erst in den RRF-Pool. Untergrenze, verhindert dass völlig irrelevante
   Vector-Hits durch FTS5-Boost reingehoben werden.
2. **Post-RRF Score-Threshold:** Final-RRF-Score muss > 0.015 sein.
   Default-Wert pragmatisch — typisch das Niveau, wo "Item in nur einer
   Source mit Rank 5+" reinfällt. ENV-Variable
   `EPISODIC_RRF_THRESHOLD` für Tuning in Phase 5.

**Final Top-K:** unverändert 3 (entspricht aktuellem `EPISODIC_TOP_K`).

## Real-Data-Befund-Mapping

Wie die Smoke-Befunde aus `docs/3.4-SMOKE.md` durch 3.4.I adressiert
werden:

**Bayreuth-Halluzinations-Szenario:**
- Vector findet 3 thematisch-falsche Hits mit Sim ~0.70-0.75
- FTS5 findet 0 Hits (keine Konversation enthält "Bayreuth"/"Marc"/
  "Parsifal" als Token — der Pending-Fact `contact_bayreuth` ist nicht
  im Memory-FTS-Index)
- **Mit RRF:** Vector-Items kriegen Single-Source-Rank. RRF-Score
  ist halbiert. Wenn alle unter Threshold 0.015 fallen → Final: 0 Hits
  → Episodic-Schicht leer → Twin sagt "ich erinnere mich nicht an eine
  Bayreuth-Konversation" (statt Halluzination).

**Maria-Workshop-Szenario (positiver Fall):**
- Vector findet Maria-Konv mit Sim 0.718
- FTS5 findet sie auch (Tokens "Maria", "Workshop", "September" matchen)
- **Mit RRF:** Hit ist in beiden Sources, kriegt vollen RRF-Score,
  rankt klar als Top-1 → wird weiter zuverlässig gefunden.

**Anna-Test-Szenario (E5-Live aus 3.4.E):**
- Vector findet "Anna ist Markus' Frau" mit Sim 0.7395
- FTS5 würde Token "Frau" matchen, aber nicht "Anna" (außer wenn Query
  "Anna" enthält)
- Bei Query "Wer ist Markus' Frau?": Sanitization → `Markus Frau`,
  FTS5 findet's, Vector findet's → RRF-Boost auf "Anna"-Item.

Konsistent mit Erwartung: korrekte Hits werden bevorzugt, Token-Overlap-
Halluzinationen werden bestraft.

## Implementation-Hinweise

**1. FTS5-Search-Pattern:**

```sql
SELECT target_type, target_id, bm25(memory_fts) AS bm25_score
FROM memory_fts
WHERE memory_fts MATCH ? AND twin_id = ?
ORDER BY bm25_score
LIMIT 10;
```

`bm25()` returnt **negativen** Score (kleiner = besser). Beim Ranking
in RRF aufpassen — Rang aus `ORDER BY bm25_score ASC`.

**2. Parallel-Search:**

Vector und FTS5 können parallel laufen (`Promise.all`). Beide sind I/O-
bound (SQLite-Calls), Parallelisierung spart ~5-10ms. Optional, aber
sinnvoll wenn einfach.

**3. RRF-Implementation:**

```typescript
function rrfMerge(
  vectorHits: SearchHit[],
  fts5Hits: SearchHit[],
  k = 60,
): RrfHit[] {
  const scores = new Map<string, RrfHit>();
  
  vectorHits.forEach((hit, idx) => {
    const id = hit.embeddingId;
    const rank = idx + 1;  // 1-indexed
    const contribution = 1 / (k + rank);
    scores.set(id, { ...hit, rrfScore: contribution, vectorRank: rank });
  });
  
  fts5Hits.forEach((hit, idx) => {
    const id = hit.embeddingId;
    const rank = idx + 1;
    const contribution = 1 / (k + rank);
    const existing = scores.get(id);
    if (existing) {
      existing.rrfScore += contribution;
      existing.bm25Rank = rank;
    } else {
      scores.set(id, { ...hit, rrfScore: contribution, bm25Rank: rank });
    }
  });
  
  return Array.from(scores.values()).sort((a, b) => b.rrfScore - a.rrfScore);
}
```

**4. Min-Vector-Sim als Pre-Filter:** im Vector-Pool vor RRF-Merge:

```typescript
const vectorPoolFiltered = vectorHits.filter(h => h.similarity >= 0.5);
```

**5. ENV-Variablen:**

- `EPISODIC_HYBRID_RRF_K=60` (Standard-RRF-Konstante)
- `EPISODIC_HYBRID_POOL_SIZE=10` (Top-K pro Source)
- `EPISODIC_HYBRID_MIN_VECTOR_SIM=0.5` (Pre-RRF-Filter)
- `EPISODIC_RRF_THRESHOLD=0.015` (Post-RRF-Cutoff)

Plus die bestehenden aus 3.4.E:
- `EPISODIC_TOP_K=3` (Final-Output bleibt unverändert)
- `EPISODIC_MIN_QUERY_LENGTH=10` (Min-Length bleibt für Skip-Logic)

**Wegfallen:**
- `EPISODIC_SIMILARITY_THRESHOLD=0.7` — wird durch zweistufiges
  Threshold-System ersetzt. Wert aus `.env` und Code rausnehmen.

## Größe und Risiko

**Größe:** M-L. Geschätzt 2-3 Stunden Bauzeit + Tests. Vergleichbar mit
3.4.E.

**Risiken:**

- **BM25-Score-Semantik in SQLite:** `bm25()` returnt negative Scores
  (kleiner = besser). Beim Sortieren aufpassen.
- **FTS5-Tokenizer-Konsistenz:** wenn `memory_fts` mit `unicode61
  remove_diacritics 2` erstellt wurde, muss Sanitized-Query *nicht*
  zusätzlich diacritics-stripped werden (Tokenizer macht's intern).
- **RRF-Threshold ist Pi-mal-Daumen:** Default 0.015 ist Schätzung,
  Phase-5-Validierung kann zeigen dass anderer Wert besser ist. ENV-
  tunable mit klarer Doku.
