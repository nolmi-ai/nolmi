## Phase 2 — Maintenance-CLI (Bestandsdaten-Embedding)

**Ziel:** Die 26 pending Konversationen embedden. Verifizieren dass
Skip-Logic (Konversationen mit Summary-Segments) und sequenzielle
Pipeline sauber laufen.

**Befehl 2.1 — Dry-Run:**

```bash
pnpm twin:memory-embed-all @markus -- --dry-run
# äquivalent: pnpm --filter @twin-lab/runtime twin:memory-embed-all @markus -- --dry-run
```

**Erwartung:** zeigt alle pending Items, kein Insert. `processed > 0`,
`succeeded = 0`, `skipped` deckt Konversationen mit Segments + alle
processed im dry-run-Modus (Service-Buchhaltung).

**Ergebnis:**

```
twin:memory-embed-all  twin=@markus
  mode:    pending/failed only
  type:    all
  DRY-RUN — keine Embeddings, keine Status-Updates

  ⊘ skip conversation/conv_jTIJJjllbAl9…
  ⊘ skip conversation/conv_TkbrNOQHrYeq…
  ⊘ skip conversation/conv_y2MOgHhDIIqg…
  ⊘  dry-run  (×23)

fertig in 0.0s
  processed:  23
  succeeded:  0
  failed:     0
  skipped:    26    (3 mit Segments + 23 dry-run-skipped)
```

**Erwartung erfüllt?** ✓ — drei Konversationen haben Summary-Segments
(werden geskipped, weil Segments ihren Inhalt schon abdecken), 23 würden
echt embedded.

**Befehl 2.2 — Real-Run:**

```bash
pnpm twin:memory-embed-all @markus
```

**Erwartung:** 23 Konversationen werden embedded (Modell-Load beim
ersten Item ggf. 20-50s). Skip-Counter zeigt die 3 Segment-Konversationen.
Keine Failures.

**Ergebnis:**

```
[1/23]–[23/23] alle ✓
fertig in 2.9s
  processed:  23
  succeeded:  23
  failed:     0
  skipped:    3
```

**Latenz:** 2.9s für 23 Items (~126 ms/Item, inkl. q8-Modell-Pipeline).
Modell war wohl noch aus Test-Lauf gestern warm — bei kaltem Start
~30-50s Mehrkosten.

**Erwartung erfüllt?** ✓ — alle 23 Konversationen embedded, 3 Skips wie im Dry-Run.

**Befehl 2.3 — Post-Run-Verifikation:**

```bash
sqlite3 -header data/twin.db "SELECT target_type, COUNT(*) FROM embeddings GROUP BY target_type;"
sqlite3 -header data/twin.db "SELECT 'conversations' AS source, embedding_status, COUNT(*) FROM conversations WHERE status='ended' GROUP BY embedding_status;"
sqlite3 data/twin.db "SELECT COUNT(*) FROM memory_fts;"
```

**Erwartung:** `embeddings.conversation = 24` (1 vorher + 23 neu),
`conversations.done = 27` (1 + 23 + 3 skip→done), `memory_fts = 24`.

**Ergebnis:**

| source                 | status | count |
| ---------------------- | ------ | ----- |
| embeddings.conversation | —     | **24** |
| conversations          | done   | **27** |
| memory_fts             | —      | **24** |

**Erwartung erfüllt?** ✓ — Counts passen exakt. Skip-Konversationen
wurden auf `done` gezogen, damit die Pending-Liste konvergiert. FTS5-
Tabelle hat Eintrag pro Embedding.
