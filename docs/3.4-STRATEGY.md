# 3.4-STRATEGY.md — Phase 3.4 Memory: Episodic

**Stand:** 12. Mai 2026 Nachmittag, nach Strategie-Session und Pre-Check Stack-Validation. Inkl. Befunden zu sqlite-vec/better-sqlite3-Implementation-Details, Name-Overlap-Limitation, FTS5-Datenschicht-Vorbereitung, q8-Quantisierung als Default.

Dieses Dokument hält die Architektur-Entscheidungen für Phase 3.4 (Memory: Episodic) fest. Es ist Bau-Begleiter zur Phase — referenziert in Sub-Schritt-Briefings, nachgehalten beim Commit, archiviert nach Phasen-Abschluss.

**Pattern für künftige Phasen:** Jede größere Phase bekommt eine eigene `<phase>-STRATEGY.md`. Strategie-Sessions resultieren in Strategy-Doc, dann Sub-Schritt-Bau, dann Commit-Sequenz.

---

## Was Episodic-Memory ist

> Twin erinnert sich an *spezifische Events* aus der Vergangenheit, abrufbar über semantische Ähnlichkeit — nicht nur "was war chronologisch", sondern "was war thematisch relevant zur aktuellen Situation".

Technisch: Text-Fragmente werden in Vektoren umgewandelt (numerische Repräsentationen semantischer Bedeutung). Bei einer neuen Frage wird auch sie zu einem Vektor, das System sucht die ähnlichsten Vektoren aus der Vergangenheit. Vector-Similarity-Search via `sqlite-vec` Extension.

---

## Foundation-Layer vs. Pattern-Layer

Aus der Vision-Session (TWIN-VISION.md) sind fünf menschliche Patterns identifiziert, die Episodic-Memory als Substrate brauchen:

- **Zeit-Erleben** — wann war was, Frequenz-Tracking
- **Schlaf/Träume** (Dream-Pattern #94) — Cross-Conversation-Verdichtung
- **Aufmerksamkeit/Fokus** — aktuelles Hauptthema des Owners
- **Lebens-Narrativ** — kohärente Geschichte aus Fragmenten
- **Selbst-Reflexion** — Twin denkt über sich selbst nach

**3.4 = Extended Foundation.** Wir bauen das Foundation-Layer plus alle Datenschicht-Erweiterungen, die für die fünf abhängigen Patterns klar antizipierbar sind. Pattern-spezifische Logic kommt mit den jeweiligen Pattern-Phasen.

**Begründung:** Konsistent mit dem heutigen "sauber bauen statt Backlog ansammeln"-Pattern (analog 3.1 DB-Storage von Anfang an, 3.2 Multi-Tenant von Anfang an). Spätere Pattern-Implementierungen sind dann Logic-Erweiterungen, keine Schema-Brüche. Mehr jetzt investieren, weniger später.

**Mitigation gegen Over-Engineering:** Pattern-Vorbereitungen sind minimal — nur die Daten-Schicht-Erweiterungen plus Repo-Methoden. Logic kommt jeweils mit dem Pattern-Implementations-Phase. Beispiel: bei "Zeit-Erleben" baut 3.4 nur `last_accessed_at`/`access_count`-Spalten plus Basis-Repo. Helpers wie "wann war die letzte Konversation über X" kommen mit Pattern-Phase.

## Architektur-Entscheidungen

### Entscheidung 1 — Embedding-Provider

**Swappable Interface mit drei Implementierungen, ENV-konfigurierbar pro twin-lab-Installation.**

```
EmbeddingProvider interface
  ├── LocalEmbeddingProvider     (Default für Self-Hosted)
  ├── OpenAIEmbeddingProvider    (Hosted SaaS, optionale System-API-Key)
  └── VoyageEmbeddingProvider    (Anthropic-nahe Setups, RAG-optimiert)
```

Configuration:
```
TWIN_LAB_EMBEDDING_PROVIDER=local | openai | voyage
TWIN_LAB_EMBEDDING_API_KEY=<key for openai/voyage>
TWIN_LAB_EMBEDDING_MODEL=<model name>
```

**Default-Werte:**
- `local` → `Xenova/multilingual-e5-large` (1024 dim, deutsch-kompatibel)
- `openai` → `text-embedding-3-small` (1536 dim)
- `voyage` → `voyage-3` (1024 dim)

**Lokales Modell — Quantisierung:**

`multilingual-e5-large` hat zwei Varianten:
- `fp32` (Original-Gewichte): ~2.1 GB Disk, beste Qualität, langsamere Inference
- `q8` (8-bit-Quantisierung): ~560 MB Disk, minimaler Qualitätsverlust bei Embeddings, schnellere Inference

**Default `q8`** für Production. Quality-Loss durch q8-Quantisierung ist bei Embeddings (anders als bei Generative-Modellen) praktisch nicht messbar. Faktor-4-Speicher-Reduktion plus schnellere Inference rechtfertigen den Default.

ENV-Konfiguration:
```
TWIN_LAB_EMBEDDING_DTYPE=q8 | fp32
```

In Transformers.js via Pipeline-Option:
```javascript
const extractor = await pipeline(
  'feature-extraction',
  'Xenova/multilingual-e5-large',
  { dtype: process.env.TWIN_LAB_EMBEDDING_DTYPE ?? 'q8' }
);
```

**Begründung der Provider-Auswahl:**

Self-Hosting-Use-Case (insbesondere Enterprise mit isolierten Umgebungen) verlangt Pattern, das ohne externe APIs funktioniert. `LocalEmbeddingProvider` als Default deckt alle Deployment-Szenarien (Hosted SaaS, Self-Hosted, Enterprise) plus respektiert User-Erwartung "ein API-Key pro Twin" (für LLM, nicht zusätzlich für Embeddings).

`OpenAIEmbeddingProvider` und `VoyageEmbeddingProvider` als optionale Modes für Power-User oder Hosted-SaaS-Default mit Quality-Upgrade. Platform-Operator zahlt Embedding-Kosten (trivial: ~$0.02/1M tokens).

**Lokales Modell — warum multilingual-e5-large:**

Twins sind deutsch. Englisch-optimierte Modelle (z.B. all-MiniLM-L6-v2, nomic-embed-text) haben schwache deutsche Performance. `multilingual-e5-large` ist substantiell besser für deutsche Inhalte. Trade-off: q8-Variante ~560 MB statt 25-140 MB der kleineren Modelle, aber Server-Deployment-tauglich.

**E5-Pattern-Hinweis:** E5-Modelle erwarten Prefix für Query vs. Passage:
- Queries: `"query: <text>"`
- Documents (Passages): `"passage: <text>"`

Das hat Konsequenzen für Embedding-Pipeline — User-Messages und gespeicherte Memory müssen mit korrektem Prefix embedded werden. Wird in EmbeddingProvider-Interface abstrahiert.

**Embedding-Model pro Vektor mitgespeichert.** Spalte `embedding_model` in der Embeddings-Tabelle. Ermöglicht:
- Re-Embedding bei Modell-Update
- Migration bei Provider-Wechsel
- Heterogene Coexistenz (alte Vektoren mit Modell X, neue mit Modell Y)

**Cross-Twin-Embedding-Search im A2A:**

Aktuell nicht in 3.4. Verschiedene Provider produzieren inkompatible Vector-Spaces. Wenn später relevant (Föderation, Phase 4.4), müssen Twins ihre Embedding-Provider abgleichen oder es gibt eine A2A-Embedding-Bridge. Backlog-Kandidat.

### Entscheidung 2 — Embedding-Granularität

**Primary: pro Summary-Segment. Plus: pro abgeschlossene Konversation falls keine Segments existieren.**

Trigger-Punkte:
- **3.3.B Summary-Generation:** wenn neuer Summary-Segment entsteht → embed das Segment
- **3.3.G3 Conversation-Reset:** wenn Konversation finalisiert wird → wenn keine Segments existieren, embed die ganze Konversation als ein Vektor; wenn Segments existieren, ignorier (sind schon embedded)

**Begründung:**

Direkte Anknüpfung an 3.3.B (Summary-Engine bereits gebaut, Output direkt verwendbar). Geringe zusätzliche Komplexität. Edge-Case "kurze Konversationen unter 50 Messages werden nie embedded" durch Conversation-Reset-Trigger abgedeckt.

**Was nicht in 3.4:**

- Audit-Event-Level-Embedding als Default (zu viele Vektoren, Storage-Grow zu groß)
- Semantic-Block-Cutting (Topic-Detection eigenes Engineering-Problem, Heuristiken ungenau)

Audit-Event-Level als spätere Erweiterung möglich für "rich layer" wenn feinere Suche gewünscht. Backlog-Kandidat falls Pattern-Phase es verlangt.

### Entscheidung 3 — Retrieval-Strategie

**Always-On Top-K=3 mit Similarity-Threshold ~0.7.**

Default-Verhalten im Send-Path:
1. User sendet Message
2. Message wird embedded (Search-Query)
3. Vector-Search über alle Episodic-Embeddings des Twins
4. Top-3 Results mit Similarity > 0.7 werden geladen
5. Results als sechste Schicht in System-Prompt eingefügt, zwischen Summaries und Live-Window
6. LLM-Call mit erweitertem Kontext

**Konfiguration via ENV:**
```
EPISODIC_TOP_K=3
EPISODIC_SIMILARITY_THRESHOLD=0.7
```

**Search-Query:**

Nur die aktuelle User-Message wird embedded (Option A aus Strategie-Session). Live-Window-Kontext-Anreicherung (Option B) als spätere Erweiterung möglich, wenn Retrieval-Qualität schlecht.

**Cost-Control:**

- Min-Token-Filter: User-Messages unter 5 Tokens werden nicht embedded ("Hallo" triggered keine Vector-Search)
- Caching: identische User-Inputs in kurzem Zeitraum nicht doppelt embedded
- Threshold-Filter: nur Results mit Similarity > 0.7 in Prompt aufgenommen (vermeidet irrelevante Memories)

**Warum nicht Tool-Call-Pattern (Strategie 4):**

Tag-10/11 hat Item #89 etabliert (Claude Opus 4.7 mit `toolChoice: 'auto'` zurückhaltend bei autonomen Tool-Calls). Episodic-Search-via-Tool hätte gleiches Problem. Tool-Picker-Pattern aus 3.2.H könnte später adaptiert werden, falls Tool-Call-basierte Memory-Abfrage gewünscht. Spätere Erweiterung.

**Warum nicht LLM-Self-Triggered (Strategie 3):**

Zwei LLM-Calls pro Send (Vor-Check + Antwort) — Latenz und Kosten verdoppelt. Pre-Check-LLM kann falsch liegen. Pragmatisch: Always-On mit Threshold ist deterministischer und einfacher.

**Warum nicht Heuristic-Triggered (Strategie 2):**

Heuristiken sind ungenau. "Erzähl mir nochmal über Marc" — kein Vergangenheits-Marker, aber Episodic wäre sinnvoll. Always-On vermeidet diese Lücken.

### Entscheidung 4 — Update-Strategie

**Synchron beim Schreiben.**

Konkret:
- Wenn 3.3.B-Summary-Generation einen neuen Segment erzeugt → embedded direkt im selben Code-Pfad
- Wenn 3.3.G3-Reset eine Konversation finalisiert (keine Segments) → embedded direkt
- Wenn manuell via CLI → embedded direkt

**Begründung:**

Embedding-Operations sind schnell (~100-400ms je nach Provider). Frequenz ist niedrig (pro Segment-Generation oder Reset, nicht pro Message). Synchrone Latenz akzeptabel weil User-Aktion eh kurze Wartezeit erwartet.

Background-Job-Infrastruktur (Strategie B aus Diskussion) explizit nicht in 3.4 — kommt eventuell mit Dream-Pattern (#94) als eigene Phase, weil dort substanzielle Background-Logic gebraucht wird.

**Failure-Handling:**

Embedding-Failure unterbricht *nicht* die Hauptoperation (Summary-Generation, Reset). Failure wird geloggt, Segment/Konversation landet mit Status-Flag `embedding_status = 'pending'`. Separate Re-Try-Logik via Maintenance-CLI.

**Wo lebt die Pending-Queue:**

Boolean-Flag auf existierenden Tabellen (Strategie B aus Diskussion). Konkret:
- `conversation_summaries.embedding_status` (text: 'done' | 'pending' | 'failed')
- `conversations.embedding_status`
- `twin_diary.embedding_status`

Einfacher als eigene Queue-Tabelle. Bei Bedarf später erweitern.

**Maintenance-CLI:**

```bash
twin:memory-embed-all <handle> [--force]
```

Funktionen:
- Ohne `--force`: nur Pending-Items embedden (für Failure-Retry)
- Mit `--force`: alle Items neu embedden, auch bereits done (für Provider-Wechsel oder Modell-Update)

### Entscheidung 5 — Pattern-Vorbereitung (Extended Foundation)

**Datenschicht-Erweiterungen für die fünf abhängigen Patterns. Logic kommt mit Pattern-Phasen.**

**Für Zeit-Erleben:**

Embeddings-Tabelle bekommt:
- `last_accessed_at` (TEXT, ISO timestamp, NULL initially)
- `access_count` (INTEGER, default 0)

Retrieval-Logic in 3.4 inkrementiert beide bei jedem Hit. Pattern-Phase nutzt sie für "wann war zuletzt was". Helpers in Repo (Stub-Implementierung):
- `getLastAccessedBefore(date)`
- `getMostAccessedTopics(timeWindow)`

**Für Aufmerksamkeit/Fokus:**

Embeddings-Tabelle bekommt:
- `topic_tags` (TEXT, JSON-Array, NULL initially)

Auto-Tagging-Logic kommt mit Pattern-Phase. In 3.4 nur Spalte und Repo-Method `setTopicTags(embeddingId, tags)`.

**Für Selbst-Reflexion:**

Neue Tabelle `twin_diary`:
```sql
CREATE TABLE twin_diary (
  id TEXT PRIMARY KEY,
  twin_id TEXT NOT NULL REFERENCES twin_profiles(id) ON DELETE CASCADE,
  content TEXT NOT NULL,
  triggered_by TEXT,  -- "scheduled" | "manual" | "post_extract"
  created_at TEXT NOT NULL,
  embedding_status TEXT NOT NULL DEFAULT 'pending'
);
```

Plus `target_type='diary_entry'` als neuer Eintragstyp in Embeddings-Tabelle. Twin-Diary-Einträge werden auch embedded (gleiche Pipeline wie Summaries).

In 3.4: nur Schema und Basis-Repo plus CLI für manuelle Einträge:
```bash
twin:diary-add <handle> "<content>"
twin:diary-list <handle>
```

Auto-Generierung von Diary-Einträgen kommt mit Pattern-Phase (Self-Reflection-Logic).

**Für Lebens-Narrativ:**

Embeddings-Tabelle bekommt:
- `narrative_thread_id` (TEXT, NULL initially)

Mehrere Embeddings können denselben `narrative_thread_id` haben, wenn Twin sie als "Teil derselben Geschichte" markiert. In 3.4 nur Spalte. Narrative-Construction-Logic kommt mit Pattern-Phase.

**Für Schlaf/Träume (#94):**

Datenschicht-Kompatibilität durch generisches `target_type`-System. Dream-Pattern als eigene Phase (braucht Background-Job-Infrastruktur, die 3.4 nicht baut).

## Schema-Erweiterungen (Migrations 017-019)

**Migration 017 — embeddings-Tabelle plus FTS5-Index:**

```sql
CREATE TABLE embeddings (
  id TEXT PRIMARY KEY,
  twin_id TEXT NOT NULL REFERENCES twin_profiles(id) ON DELETE CASCADE,
  target_type TEXT NOT NULL,        -- "summary_segment" | "conversation" | "diary_entry"
  target_id TEXT NOT NULL,
  embedding_model TEXT NOT NULL,
  embedding BLOB NOT NULL,           -- via sqlite-vec
  topic_tags TEXT,                   -- JSON array, NULL initially
  narrative_thread_id TEXT,
  last_accessed_at TEXT,
  access_count INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL,
  UNIQUE(twin_id, target_type, target_id, embedding_model)
);

CREATE INDEX idx_embeddings_twin_target ON embeddings(twin_id, target_type, target_id);
CREATE INDEX idx_embeddings_thread ON embeddings(narrative_thread_id) WHERE narrative_thread_id IS NOT NULL;

-- sqlite-vec Virtual-Table für Vector-Search
CREATE VIRTUAL TABLE embeddings_vec USING vec0(
  embedding float[1024]  -- multilingual-e5-large; ändern bei Provider-Wechsel
);

-- FTS5 Virtual-Table für Keyword-Search (Hybrid-Search-Vorbereitung)
CREATE VIRTUAL TABLE memory_fts USING fts5(
  content,                          -- der eigentliche Text (Summary, Konversation, Diary)
  target_type UNINDEXED,
  target_id UNINDEXED,
  twin_id UNINDEXED,
  tokenize = 'unicode61 remove_diacritics 2'  -- deutsch-aware (Umlaute, Akzente)
);
```

**FTS5-Begründung (Hybrid-Search-Foundation):**

Pure Vector-Search hat eine Pattern-Schwäche bei Eigennamen — Token-Match überwiegt manchmal semantische Ähnlichkeit (siehe Open Questions). Mitigation via Hybrid-Search (Vector + BM25) ist gewünscht.

In 3.4 wird FTS5-Tabelle populated (atomisch mit Vector-Embedding), aber Hybrid-Search-Logic im Send-Path *nicht* aktiviert. Pure Vector-Search bleibt der einzige Retrieval-Pfad. Wenn später Hybrid-Search nötig ist, ist die Datenschicht bereits vorbereitet — nur Logic-Layer fehlt dann.

Pattern konsistent mit Extended-Foundation-Approach: jetzt Schema, später Logic.

`unicode61 remove_diacritics 2` ist für deutsche Inhalte essentiell — Umlaute, Akzente, Groß/Kleinschreibung werden tokenisiert.

**Migration 018 — embedding_status-Spalten:**

```sql
ALTER TABLE conversation_summaries ADD COLUMN embedding_status TEXT NOT NULL DEFAULT 'pending';
ALTER TABLE conversations ADD COLUMN embedding_status TEXT NOT NULL DEFAULT 'pending';

-- Bestehende Daten als 'pending' markieren (werden via Maintenance-CLI embedded)
```

**Migration 019 — twin_diary-Tabelle:**

```sql
CREATE TABLE twin_diary (
  id TEXT PRIMARY KEY,
  twin_id TEXT NOT NULL REFERENCES twin_profiles(id) ON DELETE CASCADE,
  content TEXT NOT NULL,
  triggered_by TEXT NOT NULL,        -- "scheduled" | "manual" | "post_extract"
  created_at TEXT NOT NULL,
  embedding_status TEXT NOT NULL DEFAULT 'pending'
);

CREATE INDEX idx_diary_twin_created ON twin_diary(twin_id, created_at DESC);
```

## Implementation-Hinweise (Pre-Check-Befunde)

Aus dem Pre-Check (12. Mai 2026, Stack-Validation für sqlite-vec + @huggingface/transformers) sind drei nicht-triviale Implementation-Details aufgekommen, die in 3.4.A direkt mit eingebaut werden müssen — sonst Failures beim ersten Smoke-Test.

**1. BigInt für rowid bei vec0-Inserts**

```javascript
// FALSCH — wirft "Only integers are allowed for primary key values"
insertVec.run(id, buffer);

// RICHTIG
insertVec.run(BigInt(id), buffer);
```

Auch wenn `id` ein normaler Integer-String oder Number ist: bei vec0-Virtual-Tables muss er als BigInt gebunden werden. EmbeddingsRepo kapselt das in den Helper-Methoden, damit Sub-Schritte 3.4.D-G es nicht jedes Mal selbst handhaben müssen.

**2. Node-Buffer statt ArrayBuffer für Vector-Binding**

```javascript
// FALSCH — wirft "SQLite3 can only bind numbers, strings, bigints, buffers, and null"
insertVec.run(BigInt(id), f32Array.buffer);

// RICHTIG
const toBuf = (f32) => Buffer.from(f32.buffer, f32.byteOffset, f32.byteLength);
insertVec.run(BigInt(id), toBuf(f32Array));
```

Der `Float32Array` aus dem Embedding-Output muss in einen Node-Buffer gewrappt werden. Pattern wird in EmbeddingsRepo als Helper exportiert (`f32ToBuffer()`).

**3. CTE-Pattern für KNN+JOIN**

```sql
-- FALSCH — wirft "A LIMIT or 'k = ?' constraint is required on vec0 knn queries"
SELECT m.text, ev.distance
FROM embeddings_vec ev
JOIN embeddings m ON m.rowid = ev.rowid
WHERE ev.embedding MATCH ?
ORDER BY ev.distance
LIMIT 10;

-- RICHTIG
WITH knn AS (
  SELECT rowid, distance
  FROM embeddings_vec
  WHERE embedding MATCH ?
  ORDER BY distance
  LIMIT 10
)
SELECT m.target_type, m.target_id, m.embedding_model, knn.distance
FROM knn
JOIN embeddings m ON m.id = knn.rowid;
```

LIMIT muss DIREKT auf der vec0-MATCH-Query liegen, nicht nach dem JOIN. EmbeddingsRepo.search() kapselt das CTE-Pattern.

**4. pnpm-Caveat für native Dependencies**

`better-sqlite3` und `onnxruntime-node` (Transitive von `@huggingface/transformers`) sind native Dependencies. Nach `pnpm add` muss einmal `pnpm approve-builds --all` laufen, sonst fehlen die kompilierten `.node`-Files:

```
Cannot find module '.../better_sqlite3.node'
```

Im twin-lab-Repo: nach erstmaligem Hinzufügen der Deps approven, dann steht's in der pnpm-Workspace-Config und es funktioniert ohne weitere Intervention. Sub-Schritt 3.4.A-Briefing muss diesen Schritt vor dem Test-Run explizit nennen.

**5. Performance-Baseline (Apple M1 Max, Node 22)**

Aus Pre-Check (zur Orientierung — Production-VPS ist langsamer):

- Pipeline-Load (kalt, inkl. Download): ~48s
- Single embedding (warm): ~70ms
- Batch (3 Texte): ~120ms (~40ms/item amortisiert)
- Insert (text + embedding + vec0): ~36ms/item
- KNN-Query auf <10k Items: <1ms

Die 70ms Single-Embedding-Latenz im Send-Path sind im Twin-Latency-Budget unproblematisch (kommt vor LLM-Call, dessen Latenz Sekunden ist). Auf Production-VPS rechnen wir mit 150-250ms.

- Repo

**3.4.G — Maintenance-CLI**

## Production-Implikationen

**Embedding-Modell-Download beim ersten Boot:**

Lokales `multilingual-e5-large` ist ~560MB. Beim ersten Container-Boot mit `EMBEDDING_PROVIDER=local` muss das Modell gedownloadet werden. Optionen:

A) Modell im Docker-Image: Image-Size +560MB. Aber: Boot ohne Download.
B) Modell beim ersten Boot herunterladen, in persistent Volume cachen: Image bleibt klein, aber erster Boot dauert länger.
C) Image-Build-Time-Download: zwischen A und B.

**Vorschlag: B** — Image bleibt klein, Caching in persistent Volume (analog `/data` für DB). Erster Boot dauert ein paar Minuten extra (einmalig pro Deployment).

**Persistent-Volume-Mount für Modelle:**

```yaml
volumes:
  - type: volume
    source: twin-lab-models
    target: /app/models
```

Plus initialer Download-Hook beim Container-Start, der bei nicht-existierendem Modell den Download triggered.

**A2A-Bridge-Implikation:**

Bridge bleibt unverändert. Episodic-Memory ist Twin-intern, läuft in Runtime, nicht in Bridge.

## Open Questions / Risiken

**1. SQLite-vec Stabilität:** ✓ VALIDIERT

Pre-Check (12. Mai 2026) hat sqlite-vec 0.1.9 mit better-sqlite3 12.10.0 erfolgreich getestet. Alle drei Smoke-Tests grün (sqlite-vec-Standalone, Embedding-Pipeline, Integration). Performance auf M1 Max: <1ms für KNN-Query auf <10k Items. Skalierungs-Verhalten bei großen Datenmengen offen, aber für aktuellen Use-Case unkritisch.

**2. Embedding-Dimension-Wechsel:**

Wenn Provider gewechselt wird (z.B. von local 1024-dim auf openai 1536-dim), muss die `embeddings_vec`-Tabelle neu erstellt werden. Migration-Script muss das handhaben oder Provider-Wechsel als explizite Operation dokumentiert sein.

**3. Initial-Embedding-Time:**

Bei Production-Deploy von 3.4 sind bestehende Summary-Segments und reset-bare Konversationen alle als `embedding_status = 'pending'` markiert. Initial-Embedding via `twin:memory-embed-all @markus` kann je nach Daten-Menge dauern. Bei Markus aktuell wenige Items → paar Sekunden. Bei späteren Users mit viel Memory → ggf. Background-Job (also Migration-Trigger für Background-Infrastruktur).

**4. Multi-Provider-Coexistence:**

`embedding_model` pro Vektor erlaubt heterogene Coexistenz. Aber Vector-Search funktioniert nur innerhalb gleicher Dimension. Practical: Suche filtert auf `embedding_model = currentProvider`. Re-Embedding alter Daten erforderlich für komplette Coverage.

**5. Name-Overlap-Limitation bei Pure Vector-Search:**

Pre-Check hat eine pattern-eigene Schwäche aufgedeckt: bei Person-bezogenen Queries kann Token-Match die semantische Ähnlichkeit überlagern.

Konkretes Beispiel aus Pre-Check: Query „Wo geht Markus in Urlaub?" auf 5 Passages, von denen 4 das Token „Markus" enthalten. Die semantisch korrekte Toskana-Passage landete auf Rank 5/5, weil die anderen Passages höhere Token-Overlap-Scores hatten.

**Vier Mitigation-Optionen:**

a) **Query-Expansion** — User-Name automatisch in Passages eintragen wenn Subjekt klar (heuristisch, kann Memory verfälschen)
b) **Hybrid Search** — Vector + BM25/FTS5 parallel, Re-Rank (Best Practice in RAG, Datenschicht in 3.4 bereits vorbereitet via memory_fts-Tabelle)
c) **Higher K + LLM-Re-Rank** — z.B. K=20 statt 3, LLM sortiert (zusätzlicher LLM-Call pro Send)
d) **Strukturierte Felder** — Subject-Extraction beim Speichern, KNN-Filter (eigenes Engineering)

**Architektur-Entscheidung:** FTS5-Tabelle wird in 3.4 als Datenschicht aufgenommen und befüllt, aber Hybrid-Search-Logic ist nicht aktiv. Pure Vector-Search ist der einzige Retrieval-Pfad in 3.4. Wenn beim Real-Data-Test sich rausstellt dass Name-Overlap ein echtes Problem ist, kann Hybrid-Search (Option b) ohne Schema-Migration nachgereicht werden — Datenschicht ist da. Konsistent mit Extended-Foundation-Pattern.

**6. Modell-Größe-Klärung:**

`multilingual-e5-large` fp32 ist 2.1 GB Disk (Original-Gewichte). q8-Quantisierung reduziert auf ~560 MB bei minimalem Quality-Loss. **Default `q8`** für Production. fp32 als Power-Option via ENV (`TWIN_LAB_EMBEDDING_DTYPE=fp32`).

Im Docker-Setup ist Model-Cache als persistent Volume gemounted (sonst Re-Download bei jedem Container-Recreate).
