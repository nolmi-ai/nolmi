## TL;DR

**Die ursprüngliche #89-Diagnose ("Claude Opus 4.7 ruft Tools nicht autonom") ist falsch.**

Alle drei getesteten Setups (AI-SDK Anthropic + Extended Thinking, AI-SDK OpenAI, Raw Anthropic Messages API) rufen `scrape_webpage` autonom auf — bei identischem System-Prompt, identischem Tool-Schema, `toolChoice='auto'`.

Die echte Wurzel: **Twin-Service liest `result.toolCalls` (top-level), das in AI SDK 6 nur die Tool-Calls des _letzten_ Steps enthält.** Bei Multi-Step-Orchestrierung (call → tool-result → Synthese) sind Tool-Calls aus früheren Steps in `result.steps[i].toolCalls` versteckt. `detectPendingToolCall` (`twin-service.ts:1797`) und der Audit-Builder (`twin-service.ts:1620`) sehen sie nicht → kein Pending-Audit, Audit ohne Tool-Calls, finaler Synthese-Text wirkt wie „halluzinierter Output".

**Empfehlung:** Keinen der vier Fix-Pfade A/C/D/E aus #89 bauen. Stattdessen ein 1-Tages-Patch in `detectPendingToolCall` + Audit-Builder, der über `result.steps` iteriert.

## Echte Wurzel: result.toolCalls top-level vs. result.steps[].toolCalls

In **AI SDK 6** ist `result.toolCalls` semantisch der Tool-Calls-Array des **letzten** Steps. Bei einem typischen Multi-Step-Tool-Use:

```
step[0]: assistant → tool_use(scrape_webpage)    finishReason='tool-calls', toolCalls.length=1
step[1]: tool-result + assistant → text          finishReason='stop',       toolCalls.length=0
```

`result.toolCalls` zeigt step[1]'s Calls = 0. Aber `result.steps[0].toolCalls` enthält den Call.

Zwei Code-Stellen im Twin-Service lesen unzuverlässig top-level:

1. **`detectPendingToolCall` (`twin-service.ts:1797-1827`)** — iteriert über `result.toolCalls` und `result.toolResults`. Wenn der Approval-Marker im step[0] zurückkam und AI-SDK in step[1] weitergerannt ist, sieht die Funktion nichts → kein `McpToolApprovalRequiredError` → kein Pending-Audit → der Synthese-Text aus step[1] wird als finale Antwort an den User durchgereicht. Da step[1] auf einem Marker-„Tool-Result" basiert (`__MCP_PENDING_APPROVAL__`), kann der Text völlig daneben sein — was als „halluzinierte Tool-Outputs" wahrgenommen wird.

2. **Audit-Builder (`twin-service.ts:1620-1629`)** — `result.toolCalls.map(...)` für die `AuditToolCall[]`-Metadata. Selbst wenn der Tool-Call sauber lief, landet im Audit-Trail `[]` — der User sieht im Audit „keine Tool-Calls" obwohl Tools gerufen wurden. Konsistent mit der Beobachtung „LLM antwortet mit Tool-Inhalt aber Audit ist leer = Halluzination".

Der Followup-Pfad bei `forcedTool` (`twin-service.ts:1586-1611`) hat keinen Step-Walk und basiert auf demselben top-level-Check — gleicher Bug bei manuell forciertem Tool.

## Empfehlung

**Nicht A/C/D/E aus #89 bauen.** Statt der dort gelisteten Fix-Pfade (Code-Aufwand 2-3 Tage) reicht ein gezielter Patch:

### Patch (ca. 1 Tag inkl. Test)

1. **`detectPendingToolCall`** umstellen auf `result.steps.flatMap(s => s.toolCalls/.toolResults)` statt top-level. Marker im *frühesten* Step finden, der einen liefert, und werfen.
2. **Audit-Builder** ebenfalls über `result.steps` iterieren — alle Tool-Calls + Results in die `AuditToolCall[]`-Metadata.
3. **Optional, zur Sicherheit:** für Approval-pflichtige Tools `stopWhen` so verschärfen, dass AI-SDK nach dem ersten Tool-Call mit Marker-Result NICHT weiter synthetisiert (z.B. custom stopWhen-Predicate, das `__MCP_PENDING_APPROVAL__` im letzten tool-result erkennt). Defense-in-Depth gegen den oben skizzierten Halluzinations-Pfad.
4. **Re-Test 3.5.B** gegen den echten `hyperbrowser-approval`-Server — erwartet: Tool wird gerufen, Marker wird detected (jetzt aus step[0] sichtbar), Pending-Audit wird gebaut, User sieht Approval-Dialog statt Halluzinations-Text.

- **Audit-Tracking-Test:** Ein Smoke-Test der explizit checkt, dass nach einem Multi-Step-Tool-Use die `audit.output.toolCalls` non-empty ist. Würde diese Klasse von Bug zukünftig sofort sichtbar machen.
