# Phase 3.5 Strategy: Hyperbrowser als MCP-Skill (Foundation)

**Datum:** 14. Mai 2026 (Tag 16)
**Stack:** Hyperbrowser-MCP via `npx hyperbrowser-mcp`, eingebunden in
das etablierte MCP-Skill-System aus Phase 3.2
**Scope:** Foundation. Phase 3.6 (Computer-Use-Agent als Vision-
Pattern) wird in eigener Strategie-Session geplant.

## Kontext

Backlog-Item #27 ist die ausführliche Architektur-Entscheidung zu
Hyperbrowser (Cloud-Browser-Infrastruktur, Y Combinator backed,
Alternativen evaluiert). Dieses Strategy-Doc ist der **Bau-Plan**
für die Foundation-Phase: Hyperbrowser-MCP einbinden, Tools im
Tool-Picker verfügbar machen, Smoke-Test gegen realen Use-Case.

**Was Phase 3.5 explizit NICHT ist:**

- Keine Custom-Skills, die mehrere Hyperbrowser-Tools orchestrieren
- Keine Computer-Use-Agent-Integration (`claude_computer_use_agent`
  bleibt unbenutzt in 3.5)
- Keine Per-Twin-Cost-Tracking-Infrastruktur
- Keine UX-Pattern für lang-laufende Browser-Tasks

Diese kommen alle in **Phase 3.6** als Vision-relevanter Bau (OpenClaw-
Niveau Computer-Use-Erlebnis). 3.5 Foundation gibt die Capability,
3.6 macht das Erlebnis.

## Architektur-Entscheidungen

### Frage 1 — Scope: Foundation only

**Entscheidung:** Hyperbrowser-MCP wird via existierendem MCP-Skill-
System eingebunden. Alle 9 Tools werden vom Twin als Skills sichtbar,
keine Custom-Wrapping-Logik. Smoke-Test verifiziert Recherche- und
Daten-Extraktions-Use-Cases.

**Begründung:** Phase 3.2 hat die MCP-Foundation gebaut. Hyperbrowser-
MCP ist ein fertiges NPM-Package mit Standard-Schnittstelle. Foundation-
Schritt ist Konfiguration, kein Code-Change. Spätere Iterationen
(Custom-Skills, Computer-Use-Agent) brauchen empirische Daten aus dem
Foundation-Use, die wir jetzt erst sammeln müssen.

### Frage 2 — API-Key-Verwaltung: Per-Twin verschlüsselt

**Entscheidung:** `HYPERBROWSER_API_KEY` wird per Twin verschlüsselt
gespeichert (AES-256-GCM via `TWIN_LAB_ENCRYPTION_KEY`-Master-Key),
analog zu Per-Twin-LLM-API-Keys.

**Pattern in Spec-Datei:**

```json
{
  "env": { "HYPERBROWSER_API_KEY": "?" }
}
```

Der `"?"`-Wert triggert beim `pnpm twin:mcp-add` einen verdeckten
Prompt, das Eingegebene landet verschlüsselt in
`mcp_servers.env_json_encrypted`.

**Begründung:** Pattern ist schon im MCP-System etabliert (siehe
`mcp-servers/README.md`). Per-Twin-Verschlüsselung ist sauberer als
globale ENV-Variable, weil Cost-Tracking pro Twin in Zukunft möglich
ist (Hyperbrowser bietet Per-Account-Usage-Stats). Plus: konsistent
mit dem Multi-Tenant-Pattern aus Phase 2.5.

### Frage 3 — Hosting: Lokal in `mcp-servers/`

**Entscheidung:** Hyperbrowser-MCP läuft als Subprocess via `npx
hyperbrowser-mcp` im Twin-Lab-Container, analog zum existierenden
`everything`-Server. Cloud-Calls gehen an Hyperbrowser-Infrastruktur,
aber MCP-Server lebt lokal.

**Begründung:** Konsistenz mit etabliertem Pattern aus Phase 3.2.
Plus: lokaler MCP-Server kann Twin-Lab-spezifische ENV-Vars sehen
(verschlüsselter API-Key), ohne über Netz-Hops geleitet zu werden.

### Frage 4 — Twin-Scope: Nur @markus initial

**Entscheidung:** Hyperbrowser wird zuerst nur für @markus
registriert. @florian und @heiko bekommen on-demand, falls sie
substantiell genutzt werden.

**Begründung:**

- @markus ist primärer User mit aktiver Konversations-Historie
- @florian und @heiko haben 0 Production-Konversationen (Stand Tag 15)
- Hyperbrowser-Sessions kosten — keine Sinn, sie ungenutzt für nicht-
  aktive Twins zu provisionieren
- Mehr Twins später trivial via weiteren `pnpm twin:mcp-add`-Calls

### Frage 5 — Approval-Pattern: Server-weit ON

**Entscheidung:** `defaultRequiresApproval: true` für alle 9 Tools.
Twin braucht Owner-Approval vor jedem Hyperbrowser-Call.

**Begründung:**

- Hyperbrowser-Tools haben unterschiedliche Risiko-Profile: lesend
  (`scrape_webpage`, `crawl_webpages`, `extract_structured_data`,
  `search_with_bing`) vs. schreibend (`browser_use_agent`,
  `openai_computer_use_agent`, `claude_computer_use_agent`) vs.
  profile-management (`create_profile`, `delete_profile`)
- Bei den schreibenden Tools (Twin handelt im Web) ist Approval
  essentiell — Twin könnte Formulare absenden, Bestellungen tätigen
- Tool-spezifische Approval-Overrides sind im aktuellen
  Skill-System nicht trivial — Server-weit ON ist die einfache,
  konservative Wahl
- Bei zu hoher Friktion durch lesende Tools: später separater
  `hyperbrowser-readonly`-Server mit eigener Spec-Datei und
  `defaultRequiresApproval: false`, analog zum
  `everything`/`everything-approval`-Pattern

## Sub-Schritt-Plan

Sehr kompakt, weil Foundation-only:

### 3.5.A — Hyperbrowser-Spec-Datei

**Aufgabe:** `mcp-servers/hyperbrowser-approval.json` anlegen
analog zum Pattern aus README.md.

```json
{
  "name": "hyperbrowser-approval",
  "transport": "stdio",
  "command": "npx",
  "args": ["--yes", "hyperbrowser-mcp"],
  "env": { "HYPERBROWSER_API_KEY": "?" },
  "defaultRequiresApproval": true
}
```

**Größe:** XS · **Tests:** typecheck im Repo

### 3.5.B — Lokale Registrierung + Smoke

**Aufgabe:** Auf lokaler Twin-Lab-Instanz:

```bash
pnpm twin:mcp-add @markus mcp-servers/hyperbrowser-approval.json
# verdeckter Prompt: API-Key eingeben
pnpm twin:mcp-list @markus
```

**Smoke-Test:** Tool-Picker im Browser-Chat zeigt Hyperbrowser-Tools
mit Approval-Indikator. Test-Query an @markus: „Schau dir die
Anthropic-Homepage an und fass die wichtigsten drei Sätze zusammen."

**Erwartung:** Twin schlägt Tool-Call vor (`scrape_webpage`),
Owner-Approval-Modal, nach Approval läuft Scrape, Twin liefert
Zusammenfassung mit Quellen-Verweis.

**Größe:** S · **Tests:** Smoke-Run im Browser, Logs beobachten

### 3.5.C — Production-Deploy

**Aufgabe:** JSON-File im Repo committen, push, VPS-Repo pullen
(über existierendes Bind-Mount sofort verfügbar im Container, kein
Image-Rebuild). MCP-Add auf Production-Twin:

```bash
docker exec twin-lab-runtime node dist/scripts/mcp-add.js \
  @markus /app/mcp-servers/hyperbrowser-approval.json
# verdeckter Prompt: API-Key eingeben
```

**Verifikation:**

```bash
docker exec twin-lab-runtime node dist/scripts/mcp-list.js @markus
```

**Smoke-Test gegen Production:** identische Query wie lokal.

**Größe:** S · **Tests:** Production-Smoke

## Use-Cases für Smoke-Test

Drei realistische Test-Queries aus dem Markus-Twin-Profil:

1. **Recherche:** „Was hat Anthropic kürzlich auf ihrer Homepage
   geschrieben?" → `scrape_webpage` auf https://anthropic.com →

   gelistet. Hol mir Name plus Bio plus Foto-URL als JSON." →
   `extract_structured_data` mit Schema
3. **Web-Suche:** „Was sind die aktuellen Diskussionen zu MCP-

   `scrape_webpage` auf Top-Results

wird in Phase 3.6 mit Computer-Use-Agent gemacht — nicht in 3.5
Foundation.

## Was nach 3.5 Foundation kommt

**Phase 3.6 (Computer-Use-Agent) — separate Strategie-Session:**

Foundation gibt die Capability. Phase 3.6 macht das Erlebnis.
Substantielle Architektur-Fragen, die in 3.6-Strategie geklärt werden:

- **Latenz-UX:** wie reagiert UI während Twin 2-5 Minuten autonom
  im Browser arbeitet? Live-Updates? Session-Recording-Link aus
  Hyperbrowser?
- **Cost-Limits:** pro Task, pro Twin, pro Tag?
- **Failure-Modes:** Computer-Use-Agent halluziniert oder failt —
  Retry? User-Notification? Audit-Log?
- **Memory-Implications:** wie speichert Twin Web-Daten? Episodic-
  Memory mit Source-URLs? Persistent-Profile-Sessions?
- **Persona-Pattern:** wenn Twin Web-Daten zitiert, wie deklariert
  er die Quelle? Vision-konforme Transparenz.

**Vision-Kontext:**

OpenClaw-WoW-Niveau entsteht nicht durch Computer-Use alleine, sondern
durch Kombination aus Multi-Channel + Proaktivität + Computer-Use +
Self-extending Skills. Twin-Lab hat diese Patterns alle in Roadmap/
Backlog (Phase 4 Multi-Channel, #94 Dream-Pattern, 3.5/3.6 Computer-
Use, #28 Autonome Skill-Generierung).

Phase 3.5/3.6 ist Foundation für *einen* dieser Patterns. Der WoW-
Moment für Twin-Lab kommt vermutlich in Phase 4-Mitte, wenn mehrere
Patterns gleichzeitig sichtbar sind.

## Open Questions / Lessons Pending

- **Hyperbrowser-Pricing-Sensitivität:** ab $99/mo Basic. Bei wenig
  Use okay, bei Computer-Use-Agent-Sessions skaliert es. Real-Cost-
  Daten erst nach 3.6 Use sammelbar.
- **Tool-Approval-Friktion:** bei lesenden Tools (Scrape, Search)
  könnte Approval-bei-jedem-Call nervig werden. Falls so:
  `hyperbrowser-readonly`-Server als zweite Spec-Datei mit
  `defaultRequiresApproval: false` einführen.
- **Browser-Profile-Persistenz:** `create_profile` erlaubt
  persistente Cookies/Sessions. Use-Case-Pattern noch unklar — wird
  in 3.6 wichtig (Twin loggt sich für Markus in Plattformen ein).
